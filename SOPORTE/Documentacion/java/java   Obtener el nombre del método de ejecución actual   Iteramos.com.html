Devin Puntos591

Técnicamente esto va a funcionar... 

String name = new Object(){}.getClass().getEnclosingMethod().getName();

Sin embargo, una nueva clase interna anónima se creará durante el tiempo de compilación (por ejemplo. YourClass$1.class). Así que esto va a crear un .class archivo para cada método que implementa este truco. Además, un no utilizado instancia de objeto se crea en cada invocación, durante el tiempo de ejecución. Así que este puede ser un aceptable de depuración truco, pero viene con una sobrecarga considerable.

Una ventaja de este truco es que, getEncosingMethod() devuelve java.lang.reflect.Method que puede ser utilizada para recuperar todos los demás la información del método, incluyendo anotaciones y los nombres de los parámetros. Esto hace que sea posible distinguir entre los métodos específicos con el mismo nombre (sobrecarga de método).

Tenga en cuenta que de acuerdo con el JavaDoc de getEnclosingMethod() este truco no debe lanzar un SecurityException como las clases internas deben ser cargados con el mismo cargador de clases. Así que no hay necesidad de revisar las condiciones de acceso, incluso si un administrador de seguridad está presente.

Es necesario utilizar getEnclosingConstructor() de constructores. Durante los bloques fuera de (nombre), los métodos, getEnclosingMethod() devuelve null.

Enero de 2009:
Un código completo sería (para usar con @Bombe de advertencia en mente):

/**
 * Get the method name for a depth in call stack. <br />
 * Utility function
 * @param depth depth in the call stack (0 means current method, 1 means call method, ...)
 * @return method name
 */
public static String getMethodName(final int depth)
{
  final StackTraceElement[] ste = Thread.currentThread().getStackTrace();

  //System. out.println(ste[ste.length-depth].getClassName()+"#"+ste[ste.length-depth].getMethodName());
  // return ste[ste.length - depth].getMethodName();  //Wrong, fails for depth = 0
  return ste[ste.length - 1 - depth].getMethodName(); //Thank you Tom Tresansky
}

Utilizamos este código para mitigar el potencial de la variabilidad en el índice de seguimiento de la pila - ahora sólo llame nomMetodo util: 

 public class MethodNameTest {
    private static final int CLIENT_CODE_STACK_INDEX;

    static {
        // Finds out the index of "this code" in the returned stack trace - funny but it differs in JDK 1.5 and 1.6
        int i = 0;
        for (StackTraceElement ste : Thread.currentThread().getStackTrace()) {
            i++;
            if (ste.getClassName().equals(MethodNameTest.class.getName())) {
                break;
            }
        }
        CLIENT_CODE_STACK_INDEX = i;
    }

    public static void main(String[] args) {
        System.out.println("methodName() = " + methodName());
        System.out.println("CLIENT_CODE_STACK_INDEX = " + CLIENT_CODE_STACK_INDEX);
    }

    public static String methodName() {
        return Thread.currentThread().getStackTrace()[CLIENT_CODE_STACK_INDEX].getMethodName();
    }
}

Parece overengineered, pero tuvimos un número fijo de JDK 1.5 y estábamos un poco sorprendido de que cambió cuando nos mudamos a JDK 1.6. Ahora es el mismo en Java 7.6, pero uno nunca sabe. No es una prueba de que los cambios en este índice durante el tiempo de ejecución - pero esperemos HotSpot no lo hace mal. :-)
